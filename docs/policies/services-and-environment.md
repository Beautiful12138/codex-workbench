# 服务与环境边界

本文件说明 Workbench 如何登记服务、跨服务协作和处理外部环境。

## 服务登记

`services/registry.yaml` 记录服务名、本地路径、用途和备注。它用于恢复上下文和生成视图，不接管服务仓库的生命周期。

服务可以在 Workbench 目录内，也可以是外部绝对路径。未登记服务可以只读探索；需要长期协作、修改或任务关联时再登记。

## service_refs

- `service_refs` 表示任务相关服务。
- 它帮助恢复上下文、生成 index/recovery、发现缺失服务。
- 它不是文件路径白名单。
- 它不是修改授权。
- 写出的 service ref 必须能对应已登记服务。

修改授权来自用户请求、task stage、working_scope、风险边界和必要确认。

## 多服务协作

一个需求可以涉及多个服务，一个任务也可以引用多个服务。Codex 应先理解服务角色：

- 哪个服务是主要修改对象。
- 哪个服务只用于读取契约或验证。
- 哪个服务需要补充登记。
- 哪些外部服务不能写入。

跨服务契约、接口、数据模型、权限、配置或部署变化时，先暂停确认。

## 只读探索

只读探索可以读取用户指定服务、registry 中服务和必要相关文件。只读探索不等于授权修改。

允许的只读动作示例：

- 查看目录和文件。
- 搜索代码。
- 查看 Git 状态。
- 读取配置和日志。
- 运行不会改变状态的检查命令。

## 外部环境

以下行为必须先确认：

- 修改服务器、容器、K8s、数据库、Redis、MQ、Nacos 或对象存储。
- 修改系统环境变量、全局配置、权限、密钥、账号。
- 发布、部署、重启、批量搬动数据、删除、修复真实数据。
- 产生费用、影响他人或影响共享环境。

需要外部持久变更时，应记录 action note 或任务 evidence，并说明目标、命令、结果和回滚方式。

## Git 边界

Workbench 不自动接管外部服务仓库的 clone、branch、commit、worktree、push。需要对服务仓库做 Git 操作时，必须来自用户请求或当前任务授权，并遵守服务仓库自身规则。
