# 服务与环境边界

本文件说明 Workbench 如何看待服务仓库和外部环境。v1 面向个人本地协作，默认轻量、可恢复、少越界。

## 服务登记

- `services/registry.yaml` 登记服务 id、路径、用途和只读状态输入。
- `service status` 只做可达性和基础状态检查，不深度接管外部仓库。
- 未登记服务可以只读探索；需要持续协作时再登记。

## service_refs

- `service_refs` 表示任务相关服务，帮助恢复上下文和生成索引。
- 它不是文件路径白名单，也不是唯一可读范围；只读探索可按用户请求和实际问题读取必要服务。
- 修改登记服务前，应在任务的 `service_refs` 或 working_scope 等价说明中体现；写出的 service ref 必须能对应已登记服务。
- 如果探索发现需要补充相关服务，应更新任务语境或服务登记，而不是把原字段当硬限制。

## 外部环境

- 真实数据、部署、安全、权限、费用、不可逆操作、影响他人或共享环境时，先暂停确认。
- 默认不修改系统环境变量、全局配置、本地仓库历史或外部服务运行状态。
- 需要外部持久变更时，应先有用户授权，并留下 action note 或任务 evidence。

## Git 边界

- v1 不接管外部服务的 clone、branch、commit、worktree、push。
- Workbench 可以记录相关服务和状态，但不替用户管理服务仓库生命周期。
